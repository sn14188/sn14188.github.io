## 번들링 최적화

애플리케이션의 기능이 추가되고 다양한 라이브러리를 사용하게 되면 번들 크기는 자연스럽게 증가합니다. 이 과정에서 문제는 실제로 필요한 코드뿐 아니라, 사용되지 않는 코드까지 함께 포함된다는 것입니다.<br>
그 결과 사용자는 리소스에 접근하기 위해 필요하지 않은 코드까지 다운로드하게 되고 이는 곧 성능 저하로 이어집니다.<br>
번들링 최적화에 대해 얘기할 때 단순히 파일 크기만 줄이면 된다고 생각하기 쉽지만 실제 성능은 단순히 파일 크기 하나로 결정되지 않습니다.<br>
성능에 영향을 주는 요소는 다음과 같습니다.

- 자바스크립트를 얼마나 많이 다운로드하는가
- 코드를 얼마나 많이 실행해야 하는가
- 코드가 언제 로드되는가

즉 번들링 문제는 단순한 용량 문제가 아니라 다운로드, 실행, 로딩 타이밍이 결합된 문제라고 할 수 있습니다.<br>
이러한 레이어로 나눠보면 각각의 기술이 왜 필요한지와 언제 적용해야 하는지를 더 명확하게 이해할 수 있습니다.

## 번들링 최적화 전략

### 빌드 레이어

가장 먼저 고려해야 할 단계는 번들을 만들기 전에 코드 자체를 줄이는 것입니다. 대표적인 방법은 트리 셰이킹입니다.<br>
트리 셰이킹은 사용되지 않는 코드를 분석하고 제거하는 기술로 번들 크기를 줄이는 데에 가장 기본적인 역할을 합니다. 다만 모든 코드가 자동으로 제거되는 것은 아니기 때문에 코드 작성 방식도 중요합니다.<br>
<br>
예를 들어 아래와 같이 import 방식을 개선하는 것만으로도 번들 크기를 크게 줄일 수 있습니다.

```ts
// lodash 전체 import
import _ from "lodash";

// 필요한 함수만 import
import debounce from "lodash/debounce";
```

이 단계가 제대로 이뤄지지 않으면 이후에 코드 스플리팅이나 레이지 로딩을 적용하더라도 근본적인 최적화 효과를 얻기 어렵습니다.

### 번들 구조 레이어

코드를 아무리 줄이더라도, 모든 코드를 한 번에 로드하면 초기 로딩 성능은 여전히 느릴 수 있습니다. 따라서 번들을 여러 개로 나누는 구조적인 접근이 필요합니다.<br>
대표적인 방법이 코드 스플리팅입니다. 코드 스플리팅과 `dynamic import`를 활용하면 애플리케이션을 여러 개의 독립적인 번들로 분리할 수 있습니다.<br>
이렇게 의미 있는 단위로 나눠놓으면 이후의 레이지 로딩을 통해 필요한 시점에 로드할 수 있는 기반이 됩니다.

### 런타임 레이어

코드를 잘 나눴다면 이제는 언제 로드할지를 결정해야 합니다.<br>
번들 구조 레이어가 어떻게 나눌 것인가를 다루는 단계라면 런타임 레이어는 그 코드를 언제 불러올 것인가를 결정하는 단계인데, 이 문제를 해결하는 대표적인 방법이 레이지 로딩입니다.<br>
레이지 로딩은 초기 렌더링에 필요하지 않은 코드의 로딩을 뒤로 미루고, 실제로 필요한 시점에 불러오는 방식입니다. 이를 통해 초기 화면에서 다운로드하고 실행해야 하는 자바스크립트의 양을 줄일 수 있습니다.

- 페이지 단위 로딩: 특정 페이지에 진입할 때만 코드 로드
- 무거운 UI 컴포넌트 로딩:
  - 화면에 필요해지는 시점에 로드
  - 차트 라이브러리, 모달 내부에 들어가는 큰 컴포넌트 등
- 사용자 인터랙션 기반 기능 로딩
  - 사용자 액션이 발생했을 때 로드
  - 버튼 클릭 -> 에디터 로드
  - 다운로드 클릭 -> 파일 처리 로직 로드

### 네트워크 레이어

번들을 잘 최적화하더라도 네트워크 전송이 느리면 사용자 경험은 개선되지 않습니다. 이 단계에서는 생성된 번들을 더 빠르게 전달하는 것이 포인트입니다.<br>
대표적인 방법은 다음과 같습니다. 이를 통해 동일한 번들이더라도 더 빠르게 다운로드할 수 있습니다.

- 압축:
  - 서버에서 압축을 적용하면 자바스크립트 파일 크기를 크게 줄일 수 있습니다
  - 서버 설정만으로 적용할 수 있기 때문에, 먼저 적용해야 할 최적화 중 하나입니다
- Content Delivery Network:
  - 사용자와 가까운 서버에서 리소스를 제공해 지연을 최소화합니다
  - 또한 캐싱 기능을 함께 제공하는 경우가 많아, 트래픽 분산과 성능 개선을 동시에 얻을 수 있습니다
- 리소스 힌트:
  - `preload`, `prefetch`를 활용해 필요한 리소스를 미리 로드합니다
  - `preload`: 현재 페이지에 꼭 필요한 리소스를 우선적으로 다운로드하도록 합니다
  - `prefetch`: 이후에 사용할 가능성이 있는 리소스를 여유 시간에 미리 가져오도록 합니다

### 캐싱 레이어

마지막 단계는 이미 다운로드한 리소스를 재사용하는 것입니다.<br>
파일 이름에 해시를 적용하고 `Cache-Control` 헤더를 설정하면 브라우저가 리소스를 재요청하지 않고 캐시된 파일을 사용하도록 만들 수 있습니다.

- 파일 해시:
  - `main.js` -> `main.a1b2c3.js`
  - 해시는 번들러가 빌드 과정에서 자동으로 생성해주지만, Webpack과 같은 도구에서는 설정을 통해 직접 활성화해줘야 합니다
  - Vite와 같은 도구는 기본적으로 해시가 적용됩니다
- `Cache-Control` 헤더:
  - 브라우저가 리소스를 얼마나 오래 저장할지 결정합니다
  - 서버나 배포 환경에서 설정해줘야 합니다

위 두 가지를 함께 사용하면 파일이 변경되지 않는 한, 브라우저는 해당 리소스를 다시 다운로드하지 않습니다. 파일이 변경되면 해시가 바뀌어 새로운 파일로 인식되기 때문에 캐시와 최신 상태를 동시에 유지할 수 있습니다.

## 번들링 최적화 진단과 적용

### Lighthouse로 문제 찾기

구글 Lighthouse는 번들링 최적화를 직접 평가하지는 않지만, 성능 문제를 짐작할 수 있는 신호를 명확하게 보여줍니다.<br>
특히 `Reduce unused JavaScript` 항목은 현재 번들 중 사용되지 않는 코드의 양을 알려주며, `Est savings`를 통해 얼마나 줄일 수 있는지까지 확인할 수 있습니다.
이 값이 크다면 번들에 불필요한 코드가 많이 포함돼있을 가능성이 있습니다.<br>

### DevTools로 번들 분석하기

문제의 원인을 분석하는 데에는 크롬 DevTools를 활용할 수 있습니다.<br>
Network 탭에서는 전체 자바스크립트 파일의 크기와 로딩 흐름을 확인할 수 있어 어떤 파일이 용량을 많이 차지하는지, 언제 로드되는지를 파악할 수 있습니다.<br>
또한 Coverage 탭에서는 실제로 실행된 코드와 사용되지 않은 코드를 구분해서 확인할 수 있어, 다운로드된 코드들이 얼마나 활용되고 있는지까지 분석할 수 있습니다.

## Takeaways

1. 번들링 최적화는 단순히 파일 크기를 줄이는 것분아니라 다운로드, 실행, 로딩 시점까지 고려해야 합니다
2. 이를 위해 코드 최적화, 분할, 전송, 캐싱 전략을 함께 적용해야 합니다
