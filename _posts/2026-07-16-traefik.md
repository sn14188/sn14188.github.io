로컬 개발 환경을 구성하거나 API 요청을 프록시하는 과정에서 nginx는 익숙하게 써왔지만, Traefik은 이번에 일하면서 처음 접하게 됐습니다. Traefik이 무엇이고 어떻게 동작하는지 알아보고 nginx와 비교하며 정리해봤습니다.

## 리버스 프록시

리버스 프록시는 클라이언트의 요청을 대신 받아 적절한 백엔드로 전달하는 중개 서버입니다.<br>
클라이언트 입장에서는 리버스 프록시 하나만 보이고, 그 뒤에 여러 대의 백엔드 서버가 있더라도 이를 알 필요가 없게 됩니다.

### 리버스 vs. 포워드 프록시

클라이언트를 대신해 외부로 요청을 보내는 것이 포워드 프록시이고, 서버를 대신해 요청을 받는 것이 리버스 프록시입니다.

- 포워드 프록시:
   - 회사 내부망에서 외부 사이트에 접속할 때 거치는 사내 프록시 서버가 대표적입니다
   - 서버 입장에서는 실제 요청을 보낸 사람이 누구인지 모르고 프록시의 IP만 보게 됩니다
- 리버스 프록시:
   - 사용자가 `api.example.com`에 요청을 보내면, 실제로는 이 앞단의 nginx나 Traefik이 요청을 받아 내부의 여러 백엔드 서버 중 하나로 전달합니다
   - 사용자는 뒤에 서버가 얼마나 있는지 또 어떻게 구성돼있는지 알 수 없습니다

### nginx vs. Traefik

nginx와 Traefik 모두 리버스 프록시 소프트웨어입니다. 다음과 같은 기능들이 있습니다.

- 로드밸런싱:
   - 같은 역할을 하는 백엔드 서버가 여러 대 있을 때 요청을 나눠 분배하는 기능입니다
   - 한 서버에 과부하가 걸리거나 장애가 나도 나머지가 처리하도록 할 수 있습니다
- TLS 종료:
   - 클라이언트와의 암호화된 연결을 프록시에서 처리하고, 뒤의 백엔드로는 보통 암호화되지 않은 HTTP로 전달합니다
   - 인증서를 프록시 한 곳에만 두면 되고, 라우팅에 필요한 요청 내용도 (경로, 헤더 등) 이 지점에서 복호화해야 읽을 수 있습니다
- 라우팅:
   - 요청의 도메인, 경로, 헤더 등을 보고 어느 백엔드로 보낼지 결정합니다
   - 예를 들어 `api.example.com/users`는 사용자 서비스로, `api.example.com/orders`는 주문 서비스로 보내는 식입니다

이 중 라우팅 규칙을 누가, 언제 업데이트하는가에서 nginx와 Traefik은 차이가 있습니다.

- nginx: 라우팅 규칙을 설정 파일에 사람이 직접 작성하고 관리합니다
- Traefik: 라우팅 규칙을 각 컨테이너의 라벨에서 자동으로 읽어들입니다

## Traefik

Traefik은 처음부터 Docker, Kubernetes 같은 컨테이너 오케스트레이션 환경을 염두에 두고 만들어진 리버스 프록시입니다.

### 컨테이너

예전에는 서비스를 운영하려면 컴퓨터 1대를 통째로 준비해야 했는데, 컨테이너는 이걸 훨씬 가볍게 만든 것입니다. 컨테이너는 하나를 켜고 끄는 데 몇 초밖에 안 걸리고, 필요하면 똑같은 컨테이너를 순식간에 여러 개 복제할 수도 있습니다.

### 오케스트레이션

컨테이너가 수십, 수백 개가 되면 사람이 일일이 손으로 켜고 끄기 힘든데, Docker나 Kubernetes 같은 도구가 이걸 자동으로 관리해줍니다. 트래픽이 몰리면 컨테이너를 자동으로 더 만들고, 한가해지면 자동으로 줄이고, 컨테이너 하나가 죽으면 자동으로 새 컨테이너로 교체합니다.

### 정적 설정의 한계

컨테이너는 배포, 오토스케일링, 장애 복구로 재시작될 때마다 새로 만들어지는데, Docker나 Kubernetes는 미리 정해둔 IP 대역 안에서 그때그때 비어있는 주소를 하나 골라 새로 배정합니다. 집에서 쓰는 공유기가 연결되는 기기마다 정해진 범위 안에서 IP를 나눠주는 것과 같은 원리입니다.

예를 들어 nginx 설정에 아래처럼 결제 서비스 컨테이너의 주소를 작성한 상황을 가정해보겠습니다.

```nginx
location /payment {
    proxy_pass http://172.17.0.5:3000;
}
```

이 컨테이너가 재시작되면서 `172.17.0.9`라는 새 주소를 받으면, nginx는 이 변화를 전혀 모르기 때문에 위 설정 그대로 여전히 예전 주소인 `172.17.0.5`로만 요청을 보냅니다. 그 주소에는 이제 아무것도 없으니 요청은 계속 실패합니다.

그러면 nginx에도 이 범위를 통째로 적어두면 되지 않을까 싶을 수 있지만, 두 가지 이유로 불가능합니다.

- 문법적으로 불가능: nginx의 설정 문법 자체가 구체적인 IP와 포트만 받아들이고 범위는 지원하지 않습니다
- 지원해도 의미가 없음:
   - `/16` 범위에는 최대 65,536개의 주소가 들어갈 수 있지만, 그중 실제로 컨테이너가 떠서 요청을 처리하고 있는 주소는 단 몇 개뿐입니다
   - 필요한 건 "이 범위 어딘가에 있다"는 정보가 아니라 "지금 이 순간 정확히 어떤 주소가 살아있는가"라는 실시간 정보입니다

결국 nginx처럼 사람이 목록을 작성하는 방식은 이 실시간 정보를 알 방법이 없어서, 컨테이너가 새로 생기거나 사라질 때마다 사람이 직접 목록을 고쳐야 합니다. 하루에도 수십 번 바뀌는 환경이라면 현실적으로 감당이 안 됩니다.

### Traefik의 해결 방식

Traefik은 이 문제를 풀기 위해 만들어졌습니다. provider라고 부르는 대상을 (Docker 소켓, Kubernetes API 등) 실시간으로 지켜보다가, 컨테이너가 뜨고 내릴 때마다 그 컨테이너에 붙은 라벨을 읽어 라우팅 규칙을 자동으로 갱신합니다. 이러면 사람이 설정 파일을 고치거나 재시작할 필요가 없습니다.<br>

쇼핑몰이 세일을 시작해서 트래픽이 몰리면 Kubernetes가 결제 서비스 컨테이너를 자동으로 10개 더 만드는데, nginx라면 누군가 이 10개의 새 주소를 설정 파일에 적어넣고 재시작해야 하지만 Traefik이라면 이 10개가 생기는 순간 자동으로 알아채고 바로 요청을 나눠 보내기 시작합니다.

예를 들어 `docker-compose.yml`에서 컨테이너에 아래처럼 라벨을 붙이면 `api.example.com`으로 오는 요청을 이 컨테이너의 `3000`번 포트로 보내라는 라우팅 규칙이 만들어집니다.

```yaml
services:
  api:
    image: my-api:latest
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.api.rule=Host(`api.example.com`)"
      - "traefik.http.services.api.loadbalancer.server.port=3000"
```

라벨은 사실 Traefik만을 위한 기능이 아니라 Docker 자체가 제공하는 기능입니다. Docker는 그냥 저장만 하고, Traefik이 `traefik.`으로 시작하는 라벨만 스스로 찾아 읽고 해석합니다. 한 줄씩 보면 다음과 같습니다.

- `traefik.enable=true`: 이 컨테이너를 Traefik이 관리하도록 허용합니다 (없으면 Traefik은 이 컨테이너를 아예 무시합니다)
- `traefik.http.routers.api.rule=Host(api.example.com)`:
   - 요청의 `Host` 헤더가 `api.example.com`인 경우에만 이 컨테이너로 보내라는 규칙입니다
   - `routers` 뒤의 `api`는 이 라우터에 붙인 임의의 이름입니다
- `traefik.http.services.api.loadbalancer.server.port=3000`: 라우팅이 결정된 뒤 컨테이너 내부의 몇 번 포트로 전달할지 지정합니다

이 라벨들에는 컨테이너의 IP 주소가 없다는 점이 핵심입니다. IP는 라벨이 아니라 Traefik이 Docker에게 직접 물어봐서 얻습니다.

- 컨테이너가 뜨면 Traefik은 라벨에서 규칙을 읽고, 동시에 Docker API로 그 컨테이너의 현재 IP를 물어서 라우팅 테이블 항목을 만듭니다 (`api.example.com` -> `172.17.0.7:3000`)
- 컨테이너가 재시작돼서 새 IP를 받아도 라벨은 그대로이고, Docker에 물어보는 IP만 자동으로 갱신됩니다
- 이 과정이 전부 실시간으로 이뤄지기 때문에 설정 파일을 열어 고치거나 Traefik을 재시작할 필요가 없습니다

이 내용들을 표로 정리하면 다음과 같습니다.

| 구분 | nginx | Traefik |
|---|---|---|
| 설정 방식 | 정적 파일 (`nginx.conf`) | 동적 (라벨, API 기반 provider) |
| 설정 반영 | reload/재시작 필요 | 자동 반영 |
| 원래 목적 | 웹 서버 + 리버스 프록시 | 컨테이너 환경 전용 리버스 프록시 |
| 서비스 디스커버리 | 없음 (수동 설정) | Docker/Kubernetes 등 자동 감지 |
| 설정 위치 | 중앙 설정 파일 | 각 서비스(컨테이너) 라벨 |

두 방식 중 어느 쪽이 우월하다기보다는, 백엔드 구성이 얼마나 자주 바뀌는 환경인지에 따라 적합한 방식이 갈립니다.<br>
서버 목록이 고정적이라면 nginx의 정적 설정으로 충분하지만, 컨테이너가 수시로 뜨고 내리는 환경이라면 Traefik의 자동 감지가 관리 부담을 줄여줍니다. 특히 백엔드가 애초에 컨테이너가 아니라 고정 IP를 가진 전통적인 서버나 VM이라면, Traefik이 해결하려는 문제 자체가 존재하지 않기 때문에 nginx의 정적 설정이 오히려 더 자연스러운 선택이 됩니다.

## Takeaways

1. 리버스 프록시는 클라이언트 요청을 대신 받아 적절한 백엔드로 전달하는 중개 서버입니다
2. nginx와 Traefik 모두 로드밸런싱, TLS 종료, 라우팅 같은 기능을 제공하는 리버스 프록시 소프트웨어입니다
3. 라우팅 규칙이 자주 바뀌지 않는 환경에서는 nginx의 예측 가능한 정적 설정이, 컨테이너가 수시로 뜨고 내리는 환경에서는 Traefik의 자동 감지가 강점을 갖습니다
