이번 포스트에서는 훅의 등장 배경과 동작 방식, 그리고 훅이 기존의 문제들을 어떻게 해결하는지 살펴보았습니다.
<br><br>

## 등장 배경

### 상태 로직 재사용의 한계

웹 애플리케이션에서 마우스 위치를 추적하는 로직이 있다고 가정해보겠습니다.
사용자가 화면 위로 커서를 움직이면, 컴포넌트는 현재 마우스의 좌표를 상태로 저장하고 이를 화면에 표시합니다.

```js
class MouseTracker extends React.Component {
  constructor(props) {
    super(props);
    this.handleMouseMove = this.handleMouseMove.bind(this);
    this.state = { x: 0, y: 0 };
  }

  handleMouseMove(event) {
    this.setState({
      x: event.clientX,
      y: event.clientY,
    });
  }

  render() {
    return (
      <div style={{ height: "100vh" }} onMouseMove={this.handleMouseMove}>
        <h1>Move the mouse around!</h1>
        <p>
          The current mouse position is ({this.state.x}, {this.state.y})
        </p>
      </div>
    );
  }
}
```

이제 마우스 주변에 고양이 이미지를 따라다니게 하고 싶습니다. 이를 위해서는 마우스 좌표를 `Cat` 컴포넌트에 전달해야 하는데, 다음과 같은 방법을 생각해볼 수 있습니다.

- `Cat` 컴포넌트를 마우스 컴포넌트 내부에서 직접 렌더링합니다
- 마우스 좌표를 `props`로 전달합니다
- 렌더링에 사용할 함수를 `props`로 전달하는 `render prop` 패턴을 사용합니다
- 고차 컴포넌트로 감싸서 사용합니다

그러나 이러한 방식들은 모두 마우스 추적 로직을 재사용하기 위해 컴포넌트 구조를 변경하거나, 새로운 컴포넌트를 추가로 만들어야 한다는 한계가 있습니다.
단순히 마우스 위치를 추적하는 상태 로직만 재사용하고 싶은데도 컴포넌트 계층 구조가 함께 복잡해지게 됩니다.

### 생명주기 중심 구조의 복잡성

클래스 컴포넌트는 생명주기를 기준으로 코드를 나누기 때문에 서로 관련 없는 코드가 한 메서드에 섞이고, 반대로 하나의 기능에 해당하는 코드가 여러 메서드에 흩어지게 됩니다.

- `componentDidMount`에 데이터 요청과 이벤트 리스너 등록이 함께 작성되면서, 서로 다른 기능이 하나의 생명주기 메서드에 뒤섞이게 됩니다
- 이벤트 리스너의 등록은 `componentDidMount`에, 정리는 `componentWillUnmount`에 작성되면서 하나의 기능이 여러 생명주기 메서드로 분산됩니다

이처럼 코드가 생명주기 기준으로 분리되면 코드의 흐름을 파악하기 어렵고 수정할 때에도 여러 메서드를 오가야 하는 불편함이 생깁니다.
이러한 문제를 해결하기 위해, 생명주기 중심으로 코드를 나누는 대신 기능 단위로 컴포넌트를 구성하는 방식이 필요하게 되었습니다.

### 클래스 기반 구조의 부담

클래스 컴포넌트는 자바스크립트 언어 특성과 강하게 결합되어 있습니다. 그 결과 리액트 자체의 개념 이해는 물론 자바스크립트 문법과 실행 방식까지 함께 이해해야 했습니다.

```js
class MouseTracker extends React.Component {
  constructor(props) {
    super(props);
    this.handleMouseMove = this.handleMouseMove.bind(this); // ?
    this.state = { x: 0, y: 0 };
  }
  ...
}
```

- `this`는 호출 방식에 따라 달라지기 때문에 동작을 예측하기 어렵습니다
- 이벤트 핸들러를 등록할 때 메서드를 직접 바인딩해야 하는 등 추가적인 문법 지식이 필요합니다
- 함수 컴포넌트와 클래스 컴포넌트가 공존하면서 어떤 방식을 언제 사용해야 하는지 명확하지 않았고, 팀마다 다른 스타일이 생기기도 했습니다

또한 클래스는 자바스크립트의 동적인 특성에 의존하기 때문에 도구 친화적이지 않습니다.

- `this`와 프로토타입 기반 구조로 인해 코드의 실행 방식이 런타임에 결정돼 정적 분석이 어렵습니다
- 그 결과 코드 최소화, 트리 셰이킹, 핫 리로딩과 같은 최적화 작업이 복잡해질 수 있습니다

이렇게 클래스 기반 구조는 학습 부담도 높이고 최적화 측면에서도 불리합니다.
이를 해결하기 위해 훅은 클래스 없이도 상태와 생명주기를 사용할 수 있도록 해서, 리액트를 함수 중심 모델로 단순화하고 최적화 친화적인 구조를 지향하게 되었습니다.

## State 훅 사용하기

`useState`는 함수 컴포넌트 내부에서 상태를 선언하고 관리할 수 있게 해주는 훅입니다.

```js
import { useState } from "react";

function Example() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  );
}
```

`useState`는 두 개의 값을 반환합니다.

```js
const [count, setCount] = useState(0);
```

위 예에서 `count`는 현재 상태 값, `setCount`는 상태를 변경하는 함수입니다.
0은 초깃값이며, 버튼을 클릭해 `setCount`를 호출하면 리액트는 새로운 상태를 저장하고 컴포넌트를 리렌더링합니다. 그 결과 변경된 `count` 값이 화면에 반영됩니다.<br>
`useState`는 `[현재 상태, 상태 변경 함수]` 형태의 배열을 반환하며, 흔히 이를 구조 분해해 사용합니다.

## Effect 훅 사용하기

`useEffect`는 함수 컴포넌트에서 부수 효과를 처리할 수 있게 해주는 훅입니다.
`useEffect` 훅은 `componentDidMount`와 `componentDidUpdate`, `componentWillUnmount`의 역할을 하나의 API로 통합한 개념으로 이해할 수 있습니다.<br>
부수 효과 또는 효과란 리액트의 렌더링과는 별개로, 화면을 그리는 것 외에 추가로 수행해야 하는 작업을 의미합니다. 보통 다음과 같은 작업들이 이에 해당합니다.

- 데이터 요청
- 타이머 설정
- 이벤트 리스너 등록
- 외부 API 연동

```js
import { useEffect, useState } from "react";

function Example() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    document.title = `You clicked ${count} times`;
  }, [count]);

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  );
}
```

### 클린업이 필요 없는 Effect

어떤 작업들은 실행 이후에 되돌리거나 해제할 필요가 없습니다. 예를 들어 문서 제목 변경이나 단순한 데이터 요청이 이에 해당합니다.<br>
위 코드에서는 `count` 값이 변경될 때마다 효과가 실행됩니다.

```js
useEffect(() => {
  document.title = `You clicked ${count} times`;
}, [count]);
```

이 작업은 이전 상태를 정리할 필요가 없기 때문에 별도의 클린업 함수가 필요하지 않습니다. 매번 값을 덮어쓰기 때문입니다.

### 클린업이 필요한 Effect

어떤 효과는 단순히 값을 변경하는 것이 아니라, 외부 시스템에 구독을 등록하거나 리소스를 생성합니다.<br>
이런 경우에는 클린업이 필요합니다. 만약 클린업을 하지 않으면 컴포넌트가 사라져도 구독이 남아있을 수 있고 렌더링이 반복될 때 구독이 중복 등록될 수 있습니다.

```js
import { useState, useEffect } from "react";

function FriendStatus(props) {
  const [isOnline, setIsOnline] = useState(null);

  useEffect(() => {
    function handleStatusChange(status) {
      setIsOnline(status.isOnline);
    }

    ChatAPI.subscribeToFriendStatus(props.friend.id, handleStatusChange);

    return () => {
      ChatAPI.unsubscribeFromFriendStatus(props.friend.id, handleStatusChange);
    };
  }, [props.friend.id]);

  if (isOnline === null) return "Loading...";
  return isOnline ? "Online" : "Offline";
}
```

위 코드에서는 컴포넌트가 렌더링된 뒤 `useEffect`가 실행되고, `ChatAPI.subscribeToFriendStatus`를 통해 친구의 상태를 구독합니다.<br>
`useEffect`는 함수를 반환하면 그것을 클린업 함수로 인식합니다. 따라서 구독을 등록하는 코드와 구독을 해제하는 코드가 하나의 블록에 함께 작성됩니다.

```js
useEffect(() => {
  subscribe();
  return () => unsubscribe();
}, []);
```

이 구조 덕분에 관련된 로직이 흩어지지 않고 한 곳에서 함께 관리할 수 있습니다.

리액트는 다음 두 시점에 클린업을 실행합니다.
이 동작 덕분에 컴포넌트가 화면에 존재하는 동안에는 하나의 구독만 유지되며, 구독 중복이나 메모리 누수와 같은 버그를 방지할 수 있습니다.

- 컴포넌트가 언마운트될 때
- 효과가 다시 실행되기 직전

## 종류

리액트가 제공하는 다양한 훅들은 용도에 따라 다음과 같이 나눌 수 있습니다.

### 기본 훅

- `useState`: 상태를 선언하고 관리합니다
- `useEffect`: 렌더링 이후의 부수 효과를 처리합니다
- `useContext`: 상위에서 제공한 컨텍스트 값을 `props` 전달 없이 사용할 수 있게 해줍니다

### 추가 훅

- `useReducer`: 복잡한 상태 로직을 리듀서 패턴으로 관리합니다
- `useCallback`: 함수를 메모이제이션합니다
- `useMemo`: 계산 결과를 메모이제이션합니다
- `useRef`: DOM을 참조하거나, 값이 변경되어도 리렌더링을 발생시키지 않는 변수를 저장합니다
- `useImperativeHandle`: ref를 통해 노출할 값을 제어합니다
- `useLayoutEffect`: DOM 변경 직후 동기적으로 실행됩니다
- `useDebugValue`: 커스텀 훅 디버깅을 도와줍니다
- `useDeferredValue`: 업데이트를 지연시켜 UI 반응성을 유지합니다
- `useTransition`: 상태 업데이트를 트랜지션으로 처리합니다
- `useId`: 고유한 ID를 생성합니다

### 라이브러리 훅

- `useSyncExternalStore`: 외부 스토어와의 일관된 동기화를 지원합니다
- `useInsertionEffect`: 스타일 삽입과 같은 낮은 수준의 작업에 사용됩니다

## Takeaways

1. 훅은 상태 로직을 함수 단위로 분리하고 재사용 가능하게 만들어 컴포넌트 구조 변경 없이 로직 재사용이 가능하도록 했습니다
2. 훅은 생명주기 기준이 아닌 기능 단위로 코드를 구성할 수 있게 해 관련 로직이 흩어지는 문제를 해결했습니다
3. 훅은 클래스를 사용하지 않고 함수 중심 모델로 전환함으로써 코드를 단순화했습니다
