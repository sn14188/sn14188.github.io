인터넷 환경에서 콘텐츠는 서버에서 사용자에게 전달됩니다.
일반적으로 서버가 하나인 경우, 요청과 응답이 오가는 시간과 물리적인 거리 제약으로 인해 성능 병목이 발생할 수 있습니다.<br>
이러한 문제를 해결하기 위해 1990년대 후반에 CDN이 등장했습니다. CDN은 사용자와 가까운 위치에 서버를 분산 배치하여 높은 가용성과 성능을 제공합니다.<br>

## Content Delivery Network

### CDN 기술

CDN을 구성하는 서버들은 전 세계 여러 위치에 배포되며, 인터넷 백본을 통해 연결됩니다.<br>
CDN의 규모는 아키텍처에 따라 다르지만, 수천 개의 노드와 수만 개의 서버를 여러 원격 PoP에 배치하기도 합니다.

- 인터넷 백본:
  - 인터넷의 핵심 네트워크로 대륙과 국가를 연결하는 고속 전송망입니다
  - CDN 서버들은 이 백본을 통해 연결되어 빠르게 데이터를 전달할 수 있습니다
- 노드: 요청을 처리하는 지점을 말합니다
- 서버: 실제로 요청을 처리하는 물리적인 컴퓨터를 말합니다
- PoP:
  - Point of Presence의 약자로 특정 지역에 위치한 CDN 거점을 의미합니다 (서울 PoP 등)
  - 여러 서버가 묶여있는 단위로 지역 단위의 데이터 센터와 유사한 개념입니다

CDN은 트래픽이 많기 때문에, 구조에 따라 노드는 적지만 서버가 많은 경우도 있고 노드는 많지만 서버가 적은 경우도 있습니다.

### CDN 원리

콘텐츠 요청은 알고리즘에 의해 특정 기준을 바탕으로 최적화된 노드로 전달됩니다. 이때 선택 기준은 다음과 같습니다.

- 사용자까지의 hop 수가 가장 적은 위치
  - hop은 중간 경유지를 말합니다
  - 예를 들어 사용자가 CDN 서버에 바로 접근이 가능한 경우는 hop이 적고, 여러 네트워크를 통하게 되면 hop이 많아집니다
- 응답 시간이 가장 짧은 위치
- 서버 성능이 가장 높은 위치
- 비용을 기준으로 가장 저렴한 위치

CDN 제공업체는 여러 지역에 PoP를 구성하고 위 기준을 바탕으로 사용자에게 가장 적합한 서버가 선택됩니다.
일반적으로는 사용자와 가장 가까운 위치에 있는 서버가 선택되며, 이를 에지 서버라고 합니다.

### 관련 개념

- Content Provider Origin Server: 원본 콘텐츠를 제공하는 서버입니다
- CDN entry point: 원본 서버에서 콘텐츠를 가져오는 CDN 내부 서버입니다
- CDN Origin Shield: 트래픽이 급증하면 원복 서버를 보호하는 기능을 말합니다
- CDN 에지 서버: 클라이언트의 요청을 직접 처리하는 서버를 말합니다
- CDN footprint: CDN 에지 서버가 클라이언트 요청을 효과적으로 처리할 수 있는 지리적 범위를 말합니다
- CDN selector: 멀티 CDN 환경에서 여러 CDN 중에서 최적의 CDN을 선택하는 기능을 말합니다
- CDN offloading: P2P 환경에서 클라이언트 간 콘텐츠 전달을 지원하는 메커니즘입니다

## CDN 동작 방식

이제 실제로 요청이 어떻게 처리되는지 살펴보겠습니다.
CDN은 사용자 요청을 받아 가장 적절한 서버로 전달하고, 가능한 경우 저장된 콘텐츠를 활용해 빠르게 응답합니다.

- 요청 흐름:
  - 사용자 -> CDN 에지 서버 -> Content Provider Origin Server
  - CDN 에지 서버에서 처리할 수 없는 경우에 오리진 서버로 요청이 전달됩니다
- 캐싱:
  - CDN 에지 서버는 오리진 서버에서 가져온 콘텐츠를 저장합니다
  - 이후 같은 요청이 들어오면 저장된 데이터를 재사용합니다 -> 오리진 서버로 가는 요청 수를 줄일 수 있습니다
  - CDN 에지 서버에 요청한 콘텐츠가 이미 존재하는 경우 즉시 응답하고, 없는 경우에는 오리진 서버에서 콘텐츠를 가져와 사용자에게 전달합니다

## 번들링 최적화와 CDN

CDN은 단독으로도 성능을 개선할 수 있으며, 번들링 최적화와도 밀접한 관련이 있습니다.<br>
번들링 최적화에서는 코드 스플리팅을 통해 애플리케이션을 여러 번들로 나눠서 필요한 코드만 로드할 수 있도록 구조를 개선합니다.<br>
여기서 CDN이 없는 경우는 분리된 여러 번들이 하나의 서버로 집중됩니다.
반면 CDN이 있는 경우는 각 번들은 요청마다 최적의 서버로 전달되며 여러 서버에서 병렬 다운로드가 이뤄질 수 있습니다.

- 하나의 요청은 하나의 CDN 노드로 전달됩니다
- 번들이 여럿이라면 요청도 여러 개가 되고, 각각 다른 노드로 분산될 수 있습니다
- CDN은 요청마다 네트워크 상태, 캐시 여부, 서버 부하 등을 기준으로 가장 유리한 노드를 선택합니다

### 성능 관점에서의 CDN

CDN은 다음과 같은 방식으로 성능을 개선합니다.

- Latency 감소: 사용자와 가까운 서버에서 응답하기 때문에 네트워크 지연이 줄어듭니다
- TTFB 감소:
  - 요청을 보낸 뒤 첫 데이터가 도착하기까지 걸리는 시간을 말합니다
  - CDN을 사용하면 오리진 서버까지 가지 않는 경우가 많기 때문입니다
- 병렬 다운로드 효율: 여러 서버에서 동시에 리소스를 전달할 수 있습니다
- 체감 성능 개선: 초기 렌더링이 더 빨라지고, 인터랙션까지의 시간도 줄어듭니다

사용자는 더 빠르게 첫 화면을 보게 되고, 전체적인 사용 경험이 부드러워집니다.

## 실무 관점에서의 CNN

우리가 CDN 자체를 구축하지는 않지만, 번들링 결과와 리소스 구조를 통해 CDN의 효과를 극대화할 수 있습니다.

- 번들 구조 설계:
  - 코드 스플리팅을 통해 번들을 적절한 단위로 나눕니다
  - 자주 변경되지 않는 코드를 분리하면 캐시 활용도가 높아집니다
- 정적 리소스 분리:
  - HTML에서 CDN URL을 통해 리소스를 로드하도록 설정합니다
  - 자바스크립트, CSS, 이미지, 폰트 등을 CDN 경로로 제공되도록 구성합니다
- 캐싱 전략 고려하기:
  - 이름이 변경되지 않은 파일은 CDN과 브라우저에서 계속 재사용됩니다
  - 파일 이름에 해시를 포함해 변경 여부를 명확하게 합니다
- 균형 맞추기: 너무 작은 단위로 쪼개면 요청 수가 늘어날 수 있기 때문에 균형이 필요합니다

### 번들 구조 설계

페이지 단위 코드 스플리팅, 라이브러리 분리, 공통 코드 분리를 사용할 수 있습니다.<br>
예를 들어 아래 코드에서는 `HomePage.js`와 `Dashboard.js`가 모두 초기 번들에 포함됩니다. 사용자가 대시보드를 보지 않더라도 코드가 다운로드됩니다.

```js
import HomePage from "./HomePage";
import Dashboard from "./Dashboard";

function App() {
  return (
    <div>
      <HomePage />
      <Dashboard />
    </div>
  );
}

export default App;
```

아래와 같이 dynamic import를 사용하면 각 모듈이 별도 번들로 생성됩니다. 결과적으로 필요할 때 대시보드 번들이 로드되면서 초기 로딩이 빨라집니다.

```js
import React, { Suspense, lazy } from "react";

const HomePage = lazy(() => import("./HomePage"));
const Dashboard = lazy(() => import("./Dashboard"));

function App() {
  return (
    <Suspense fallback={<div>Loading...</div>}>
      <HomePage />
      <Dashboard />
    </Suspense>
  );
}

export default App;
```

### 정적 리소스 분리

프런트엔드 개발자는 자바스크립트, CSS, 이미지, 폰트 같은 정적 리소스를 CDN 경로로 분리해 제공할 수 있습니다.<br>
단순히 URL을 바꾸기 전에, 먼저 CDN에 파일을 배포하고 빌드/참조 경로를 맞추는 단계가 필요합니다.

먼저 HTML에서 CDN 경로로 로드하는 방법이 있습니다.

```html
<link rel="stylesheet" href="https://cdn.example.com/styles/main.css" />
<script src="https://cdn.example.com/scripts/app.js" defer></script>
<img src="https://cdn.example.com/images/logo.png" alt="Logo" />
```

또 Webpack, Vite 등의 빌드 도구를 사용하면 자바스크립트/CSS 번들이 자동으로 CDN URL로 참조됩니다. 그러면 번들링과 코드 스플리팅 결과를 그대로 CDN으로 연결할 수 있습니다.

```js
module.exports = {
  output: {
    publicPath: "https://cdn.example.com/", // 빌드 결과물을 CDN URL로 배포
  },
};
```

CDN URL을 활용하면 브라우저와 CDN 캐시를 활용할 수 있습니다. 그러면 사용자 위치에 따라 가장 가까운 CDN 서버에서 다운로드되므로 로딩 속도가 개선됩니다.

```css
@font-face {
  font-family: "MyFont";
  src: url("https://cdn.example.com/fonts/myfont.woff2") format("woff2");
}

body {
  background-image: url("https://cdn.example.com/images/bg.jpg");
}
```

## Takeaways

1. CDN과 번들링 최적화는 별개지만 함께 사용하면 초기 로딩과 병렬 다운로드 효율을 높일 수 있습니다
2. 프런트엔드 개발자는 번들 구조 설계, 정적 리소스 분리, 캐싱 전략을 통해 CDN 효과를 극대화할 수 있습니다
