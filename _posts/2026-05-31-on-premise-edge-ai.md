## Overview

AI 모델을 운영하는 방식은 크게 세 가지로 나눌 수 있습니다.

- 클라우드:
  - AWS, GCP, Azure 등 외부 사업자가 운영하는 서버에서 모델을 호스팅하고, 개발자는 API 호출로 AI 기능을 가져다 쓰는 방식입니다
  - 인프라를 직접 소유하거나 관리할 필요 없이 API 키만으로 시작할 수 있어 진입 장벽이 낮습니다
  - OpenAI, Claude, Google Gemini 등이 대표적입니다
- 온프레미스:
  - 조직 내부의 서버나 데이터센터에 AI 모델을 직접 설치해 운영하는 방식입니다
  - 데이터가 외부로 나가지 않아 프라이버시와 규정 준수가 중요한 환경에 적합합니다
  - Ollama, vLLM 같은 도구로 사내 서버에서 오픈소스 모델을 운영하는 것이 대표적입니다
- Edge AI:
  - 클라우드가 아닌 네트워크 에지에서 (데이터 발생 지점 근처) 추론하는 방식의 총칭입니다
  - 공장 게이트웨이, 기지국 서버, 사용자 단말기 등 다양한 형태가 있으며, 그 중 온디바이스가 가장 대표적입니다
    - 공장 게이트웨이: 공장 현장의 센서·카메라와 외부 네트워크 사이에서 데이터를 중계하는 로컬 장치입니다
    - 기지국 서버: 통신 기지국에 설치된 소형 서버로, 자율주행·스마트 신호등 같은 실시간 처리에 활용됩니다
  - 온디바이스 주요 사례:d
    - Apple Intelligence: 아이폰/맥의 Apple Neural Engine(NPU)에서 실행되는 온디바이스 AI입니다
    - Google Gemini Nano: Pixel 기기에 탑재된 경량 모델로, 요약·스마트 리플라이 등이 기기 안에서 처리됩니다
    - Samsung Galaxy AI: 갤럭시 기기에서 통화 통역, 사진 편집 등을 온디바이스로 처리합니다

클라우드는 빠르게 시작할 수 있지만, 데이터를 외부 서버로 전송해야 하고 사용량에 따라 비용이 선형으로 증가합니다.<br>
온프레미스와 Edge AI는 이 두 가지 한계를 해결하는 대안으로 주목받고 있고, 오픈소스 모델의 등장으로 이 흐름은 더욱 가속화되고 있습니다.

- Llama (Meta): 가장 널리 쓰이는 오픈소스 LLM입니다
- Mistral (Mistral AI): 비교적 작은 크기에 성능이 좋아 온프레미스에 적합합니다
- Gemma (Google): 경량 모델로 온디바이스에서도 실행 가능한 수준입니다

이 모델들이 클라우드 API 모델에 근접하거나 특정 태스크에서 앞서는 성능을 보이면서, 프라이버시나 비용을 이유로 직접 운영을 선택하는 조직이 늘고 있습니다.

## On-Premise AI

온프레미스에서는 서버, 네트워크, 스토리지 등 모든 인프라를 조직이 직접 소유하고 운영합니다.<br>
클라우드가 사업자에게 인프라 관리를 위임하는 구조라면, 온프레미스는 그 책임 전체가 조직 내부에 있습니다.<br>
구성은 크게 세 층으로 이루어집니다.

- 하드웨어:
  - LLM 추론에는 GPU가 필요합니다
  - NVIDIA A100, H100 같은 고사양 GPU 서버를 도입하거나, 소규모라면 RTX 4090 수준의 워크스테이션으로도 시작할 수 있습니다
- 모델:
  - Llama, Mistral 같은 오픈소스 모델의 가중치를 내려받아 서버에 올립니다
  - 필요에 따라 조직 내부 데이터로 파인튜닝할 수도 있습니다
- 서빙 레이어:
  - 모델을 HTTP API 형태로 노출하는 추론 서버입니다
  - vLLM은 OpenAI API와 동일한 인터페이스를 제공해, 기존에 OpenAI API를 호출하던 코드를 서버 주소만 바꿔 온프레미스로 전환할 수 있습니다

이 구성이 갖춰지면 조직 내부 네트워크에서만 접근 가능한 AI 엔드포인트가 만들어집니다. 외부로 데이터가 나가지 않는 구조 자체가 이 방식의 핵심입니다.

### 장점

- 데이터 프라이버시:
  - 데이터가 조직 내부 네트워크 밖으로 나가지 않습니다
  - 환자 정보, 금융 데이터처럼 외부 전송이 허용되지 않는 민감한 데이터를 안전하게 처리할 수 있습니다
- 규정 준수:
  - GDPR, HIPAA처럼 외부 서버 전송을 제한하는 규정을 준수할 수 있습니다
  - 데이터 주권이 강조되는 국가나 산업군에서 특히 중요합니다
- 장기 비용 절감:
  - 클라우드는 호출량에 따라 비용이 선형으로 증가하지만, 온프레미스는 초기 투자 이후 추가 비용이 거의 없습니다
  - 사용량이 많을수록 총 소유 비용 측면에서 유리해집니다
- 커스터마이징:
  - 모델 가중치에 직접 접근할 수 있어 내부 데이터로 파인튜닝하거나 추론 파이프라인을 자유롭게 구성할 수 있습니다
  - 클라우드 API에서는 불가능한 부분입니다
- 네트워크 독립성:
  - 인터넷이 불안정하거나 폐쇄망 환경에서도 안정적으로 운영할 수 있습니다

### 단점

- 초기 비용:
  - GPU 서버 등 하드웨어 도입에 상당한 초기 투자가 필요합니다
  - 최신 LLM을 원활하게 실행하려면 고사양 GPU가 필요해 진입 장벽이 높습니다
- 유지보수 부담:
  - 하드웨어 관리, 보안 패치, 모델 업데이트, 장애 대응 등을 직접 담당해야 합니다
  - 클라우드에서는 사업자가 처리해주는 일들을 내부에서 모두 책임져야 합니다
- 확장성 한계:
  - 트래픽이 갑자기 증가할 때 클라우드처럼 유연하게 확장하기 어렵습니다
  - 수요에 맞춰 서버를 추가 도입하는 데 시간이 걸립니다
- 전문 인력 필요:
  - 인프라를 운영하고 모델을 관리할 수 있는 MLOps, DevOps 역량이 내부에 있어야 합니다
- 모델 최신화:
  - 클라우드 API는 사업자가 모델을 자동으로 업데이트하지만, 온프레미스는 새 모델이 나올 때마다 직접 도입 여부를 판단하고 배포해야 합니다

### 주요 사례

- 의료 기관: 환자의 진료 기록, 영상 데이터를 외부로 전송하지 않고 AI로 분석해야 하는 병원, 제약사
- 금융사: 금융 거래 내역, 고객 계좌 정보를 다루는 곳에서 내부 AI 어시스턴트나 이상 탐지 시스템을 운영할 때
- 공공 기관: 국방, 정보기관처럼 인터넷과 단절된 폐쇄망에서 AI를 운영해야 하는 환경
- 기업 내부 도구: 사내 코드베이스, 내부 문서를 학습시킨 AI 어시스턴트를 외부 API 없이 운영하고 싶은 경우

## Edge AI

Edge AI는 클라우드가 아닌 네트워크 edge, 즉 데이터가 발생하는 지점 근처에서 AI 추론을 수행하는 방식의 총칭입니다.<br>
그중 온디바이스는 사용자의 단말기 자체에서 서버 통신 없이 기기의 CPU, GPU, NPU로 추론을 수행하는 형태로, Edge AI의 가장 대표적인 사례입니다.
온디바이스 기기에는 대형 모델을 그대로 올릴 수 없기 때문에, 아래와 같은 경량화 기법을 거쳐 크기와 연산량을 줄인 모델을 사용합니다.

- 양자화:
  - 모델의 숫자 정밀도를 낮춥니다
  - 32비트 소수점을 8비트로 줄이면 파일 크기와 연산량이 크게 감소합니다
- 프루닝:
  - 영향이 작은 파라미터를 제거해 모델을 가볍게 만듭니다
- 지식 증류:
  - 큰 모델이 작은 모델을 가르치는 방식입니다
  - 학생 모델은 선생님 모델의 풀이 과정을 보며 학습해, 혼자 처음부터 공부하는 것보다 더 좋은 성능을 낼 수 있습니다
  - 예를 들어 GPT-4 같은 초대형 모델을 기기에 올릴 수는 없지만, 그 모델에게 배운 작은 모델은 올릴 수 있게 됩니다

### 장점

- 완전한 오프라인 동작:
  - 네트워크 연결 없이도 AI 기능을 사용할 수 있습니다
  - 터널, 비행기, 인터넷이 불안정한 환경에서도 동작합니다
- 낮은 latency:
  - 서버 왕복이 없기 때문에 응답이 빠르고 실시간 처리가 가능합니다
  - 음성 인식, 카메라 필터처럼 즉각적인 반응이 중요한 기능에서 유리합니다
- 강한 프라이버시:
  - 사용자의 입력, 사진, 음성이 서버로 전송되지 않습니다
  - 세 방식 중 프라이버시를 가장 강하게 보장할 수 있습니다
- 서버 비용 없음:
  - 추론이 기기에서 일어나므로 사용자가 늘어나도 서버 인프라 비용이 증가하지 않습니다
- 개인화:
  - 사용자 데이터가 기기 안에 머무르기 때문에 외부 서버로 보내지 않고도 개인화된 모델을 유지할 수 있습니다

### 단점

- 모델 크기 제한:
  - 기기의 메모리·연산 능력에 맞게 모델을 경량화해야 하므로 클라우드 대비 성능이 제한됩니다
  - GPT-4 수준의 대형 모델을 기기에서 실행하는 것은 현재 기술로는 어렵습니다
- 기기 의존성:
  - 하드웨어 사양에 따라 실행 가능한 모델과 속도가 달라집니다
  - 저사양 기기에서는 지원하지 않거나 현저히 느릴 수 있습니다
- 업데이트 관리:
  - 모델을 업데이트하려면 앱 업데이트나 별도의 다운로드 과정이 필요합니다
  - 클라우드 API처럼 서버 측에서 일괄 배포할 수 없습니다
- 배터리 및 발열:
  - 연산이 집중되어 배터리 소모가 늘고 발열이 발생할 수 있습니다
  - 지속적으로 실행되는 기능에서는 사용자 경험에 영향을 줄 수 있습니다
- 보안 위협:
  - 모델이 기기에 저장되면 리버스 엔지니어링 시도에 노출될 수 있습니다
  - 클라우드 API와 달리 물리적 접근이 가능합니다

### 주요 사례

- 스마트폰 카메라: 얼굴 인식, 피사체 분리, 실시간 번역 등 즉각적인 반응이 필요한 기능
- 음성 비서: Siri, Google Assistant의 wake-word 감지처럼 항상 대기 상태로 동작하는 기능은 서버 요청 없이 기기에서 처리됩니다
- 자율주행: 도로 상황을 실시간으로 판단해야 하므로 서버 응답을 기다릴 수 없어 온디바이스 추론이 필수입니다
- 산업용 기기: 공장 자동화, 품질 검사 등 인터넷이 연결되지 않은 현장
- 웨어러블: 스마트워치처럼 배터리와 네트워크가 제한적인 환경의 기기

## Takeaways

1. 클라우드, 온프레미스, Edge AI는 각각 편의성, 보안, 독립성을 서로 다르게 트레이드오프합니다
2. 온프레미스는 데이터 프라이버시와 규정 준수가 중요한 B2B, 공공 영역에서 강점을 가집니다
3. Edge AI는 낮은 latency와 오프라인 동작이 필요한 모바일, 산업 환경에 적합하며, 그 중 온디바이스가 가장 널리 쓰이는 형태입니다
