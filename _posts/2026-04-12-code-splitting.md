## 코드 스플리팅

일반적인 번들링에서는 애플리케이션의 모든 코드가 하나의 파일로 묶이기 때문에 사용자가 실제로 접근하지 않는 기능의 코드까지 함께 다운로드됩니다.
이 과정에서 초기 로딩 시 불필요한 네트워크 비용과 실행 비용이 발생합니다.<br>
코드 스플리팅은 하나의 자바스크립트 번들을 여러 개의 단위로 분리해 필요한 시점에 개별적으로 로드할 수 있도록 하는 방식입니다.<br>
이를 통해 초기 로딩에 필요한 코드만 먼저 다운로드하고, 나머지 코드는 이후의 시점으로 미룰 수 있어서 결과적으로 초기 번들 크기를 줄이고 불필요한 리소스 다운로드를 방지할 수 있습니다.

### 스플리팅 기준

무조건 나누는 게 좋은 것은 아닙니다. 중요한 것은 어디서 나눌 것인지이며 이는 사용 시점을 기준으로 판단할 수 있습니다.<br>
다음 기준을 통해 통해 분리 여부를 결정할 수 있습니다.

- 라우트 기준:
  - 페이지 단위로 나누는 가장 기본적이면서 효과적인 방법입니다
  - 페이지에 진입할 때만 코드가 로드되도록 분리할 수 있습니다
- 초기 렌더링에 필요한가:
  - 초기 화면에 반드시 필요하지 않은 코드는 분리할 수 있습니다
  - 에디터, 차트 등 무거운 컴포넌트
- 모든 사용자가 사용하는가:
  - 일부 사용자만 사용하는 기능은 분리하는 것이 유리합니다
  - 설정 페이지, 관리자 기능 등
- 사용자 액션 이후에만 필요한가:
  - 사용자 행동 이후에만 필요한 기능은 초기 번들에서 제외할 수 있습니다
  - 다운로드 기능, 버튼 클릭 후 열리는 에디터 등

### Dynamic Import

코드 스플리팅은 번들러가 알아서 해주는 기능처럼 보이지만 실제로는 `import()`를 기준으로 동작합니다.<br>
즉 어디서 코드를 나눌지는 개발자가 직접 결정하게 됩니다.<br>
예를 들어 다음 코드에서는 빌드 시점에 `Editor` 코드가 메인 번들에 포함됩니다.

```js
import Editor from "./Editor";

function App() {
  return <Editor />;
}
```

그런데 다음과 같이 동적으로 `import()`를 사용하면 `Editor`가 별도 파일로 분리돼 초기 번들에서 제외할 수 있습니다.

```js
import("./Editor");
```

실제 사용 예시는 다음과 같습니다.
버튼 클릭 등으로 함수를 실행하면 그때 네트워크 요청이 발생하고 `Editor` 코드를 다운로드하고 실행하게 됩니다.
`import()`는 `Promise`를 반환하기 때문에 `await`를 써서 비동기적으로 처리할 수 있습니다.

```js
async function openEditor() {
  const module = await import("./Editor");
  module.open();
}
```

### 번들 구조

코드 스플리팅을 적용하면 하나의 번들이 여러 개의 파일(청크)로 분리됩니다.<br>
코드 스플리팅 적용 전에는 모든 코드가 하나의 번들에 포함됩니다.

```bash
dist/
 └── main.js
```

코드 스플리팅을 적용하면 번들이 다음과 같이 나뉩니다.<br>
`main.js`는 초기 로딩 시 다운로드되고, `Editor.chunk.js`는 `openEditor`가 실행될 때 로드됩니다.

```bash
dist/
 ├── main.js
 └── Editor.chunk.js
```

여기서 `Editor.chunk.js`는 개념적인 이름입니다. 실제 번들러는 보통 해시 기반의 파일명을 사용합니다.<br>
파일 내용이 변경되면 해시가 바뀌고, 변경되지 않으면 캐시를 재사용할 수 있도록 설계된 방식입니다.

```bash
dist/
 ├── main.abc123.js
 └── 245.def456.js
```

또한 여러 청크에서 동일한 라이브러리를 사용하는 경우, 번들러는 이를 공통 청크로 자동 분리하기도 합니다.

```bash
dist/
 ├── main.js
 ├── Editor.chunk.js
 ├── vendor.chunk.js
 ├── ...
```

## Lazy Loading

코드 스플리팅을 통해 번들을 나눴다면 이제는 그 코드를 언제 로드할지를 결정해야 합니다.<br>
레이지 로딩은 초기 렌더링에 필요하지 않은 코드를 나중으로 미루고, 실제로 필요한 시점에 로드하는 방식입니다.

### `React.lazy`와 `Suspense`

리액트에서는 `React.lazy`와 `Suspense`를 활용해 레이지 로딩을 구현할 수 있습니다.

- `React.lazy`는 컴포넌트를 동적으로 불러옵니다
- `Suspense`는 로딩 중일 때 보여줄 UI를 담당합니다

아래 구조에서 `Editor` 컴포넌트는 렌더링이 시도되는 시점에 로딩이 시작되며, 다운로드가 완료될 때까지는 `Suspense`의 fallback UI가 대신 표시됩니다.

```js
import React, { Suspense } from "react";

const Editor = React.lazy(() => import("./Editor"));

function App() {
  return (
    <Suspense fallback={<div>Loading...</div>}>
      <Editor />
    </Suspense>
  );
}
```

## 구현 시 주의사항

### 너무 작은 단위로 나누지 않기

코드를 지나치게 잘게 나누면 요청 수가 증가하고 네트워크 오버헤드가 발생합니다. 사용자 관점에서 의미 있는 기능 단위로 분리하는 것이 중요합니다.

### `Suspense` 범위를 적절히 나누기

아래와 같이 범위를 크게 잡으면, 일부 컴포넌트가 로딩되는 동안 화면 전체가 fallback UI로 대체됩니다.
필요한 컴포넌트 단위로 범위를 나누는 것이 좋습니다.

```js
<Suspense fallback={<Spinner />}>
  <App />
</Suspense>
```

다음과 같이 작성하면 `Header`와 `Sidebar`는 즉시 렌더링되고, `Editor` 자리에만 로딩 중임을 표시할 수 있습니다.

```js
function App() {
  return (
    <div>
      <Header />
      <Sidebar />

      <Suspense fallback={<Spinner />}>
        <Editor />
      </Suspense>
    </div>
  );
}
```

### fallback UI

다뤘던 예시처럼 단순히 로딩 중이라는 텍스트만 사용하면 레이아웃을 깨뜨리거나 화면이 깜빡이는 원인이 될 수 있습니다.

```js
fallback={<div>Loading...</div>}
```

Skeleton UI처럼 기존의 레이아웃을 유지하는 방식이 더 자연스럽습니다. Skeleton UI는 실제 콘텐츠가 로드되기 전에 동일한 레이아웃의 뼈대 형태를 미리 보여주는 로딩 방식입니다.

## Takeaways

1. 코드 스플리팅에서 `import` 구조로 번들 구조를 결정할 수 있고, 레이지 로딩으로 실제 로딩 시점을 제어할 수 있습니다
2. 잘못 나누면 오히려 성능이 나빠질 수 있기 때문에 사용 시점을 기준으로 잘 분리해야 합니다
